## 🧭 Фаза 1: Філософія та Абстракція (The Seed)

**Місія:** Управління проєктами як код (Project Management as Code). Ми відмовляємось від зовнішніх трекерів задач (Jira, Trello). Проєкт — це набір технічних **контрактів**, які проходять шлях від наміру до імплементації, забезпечуючи реальний статус прогресу автоматично.

**Базова Абстракція:** Пакет відповідає за логіку та CLI, яка читає Markdown, генерує контракти (`.spec.js`), а потім закриває їх як тести (`.test.js`) з подальшою вічною регресією (тести копіюються в кодову базу).

**Глосарій:**

- **Release Document**: `release.md` з ТЗ та відомостями про команду.
- **WIP Контракт**: `*.spec.js`, який падає/у стані `todo`, доки задачу не взяли і не зробили. Оновлює прогрес автоматично при виконанні.
- **Закритий Контракт**: `*.test.js` (результат команди `close`). Він стає законом і гарантує регресію через `pnpm test`.

## 📐 Фаза 2: Доменне Моделювання (Data-Driven Models)

Сутності і моделі, які керують процесом релізу, не мають жорсткої прив'язки до вебу, але описуються як чисті об'єкти:

- `Release`: Координує стан (версія, час початку, час завершення).
- `ReleaseDocument`: Структурує Markdown у модель (дістає задачі).
- `Person`: Типізована перевірка того, хто схвалив дії (через GPG або JSON підпис).
- `Scanner`: Абстракція доступу до файлової системи (`releases/`).

Статуси відстежуються динамічно через статус тестів: `it.todo` -> `it.skip` -> `it()`.

## 🛠 Фаза 3: Верифікація Логіки (CLI-First)

Уся бізнес-логіка @nan0web/release 100% покривається Unit-тестами та CLI командами. Жодного веб-коду. Використовуємо Message-based архітектуру `@nan0web/ui-cli`.

1. **Команди CLI (`bin/nan0release.js`):**
   - `deps`, `publish` (реалізовані).
   - Заплановані (`init`, `spec`, `close`, `status`) інкапсулюють керування `Scanner` та `Release` моделями.
2. **Contract Lifecycle (Автоматизація прогресу):**
   - На старті `.spec.js` мають `it.todo()` → задача в беклозі.
   - Коли розробник бере задачу → генерує логіку, змінює на `it()`. При коміті реальний прогрес проекту видно "зеленим".
   - `nan0release close` перейменовує `spec → test` і переносить його в `src/releases/` для 100% автоматичної регресії.

_(Для цього пакету застосовується опціонально, оскільки тут CLI-First)_

- Dashboard релізу (таблиці статусів `wip/closed/mixed`).
- Для цього ми генеруємо `.datasets/README.dataset.jsonl` за допомогою `test:docs`, щоб створювати машиночитабельні знання для LLM-агентів для донавчання або можна користуватись README.md як індексу у реальному часі.
